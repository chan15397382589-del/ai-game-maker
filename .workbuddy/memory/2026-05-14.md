## 修复续：RLS 42501 错误持续出现

### 问题
修复后用户仍报 `new row violates row-level security policy for table "users" (code:42501)`

### 诊断过程
1. 发现 `.env.local` 中 `SUPABASE_SERVICE_ROLE_KEY` 重复定义两次，两个值签名不同（差一个字母 `n` vs `b`）
2. 修复 `.env.local` 去掉重复项，保留正确的 key
3. 但用户仍报错 → 创建 `/api/debug-auth` 诊断接口
4. 诊断结果：fetch 和 JS client 用 service role 都能正常读取 users 表
5. **根因确认**：`admin-auth.ts` 等文件在**模块加载时**（`const supabaseAdmin = createClient(...)`）创建 client，环境变量在服务器启动时就绑定了，后续 `.env.local` 修改不会影响已创建的 client

### 修复方案
将所有 API 文件中的 `const supabaseAdmin = createClient(...)`（模块级创建）改为 `function getDB() { return createClient(...) }`（每次请求时创建），彻底消除环境变量缓存问题。

### 修改的文件
- `src/lib/admin-auth.ts`：`getAdminClient()` 函数替代模块级 client
- `src/app/api/admin/students/route.ts`：`getDB()` 函数
- `src/app/api/admin/projects/route.ts`：`getDB()` 函数
- `src/app/api/admin/messages/route.ts`：`getDB()` 函数
- `src/app/api/projects/route.ts`：`getDB()` 函数
- `src/app/api/save-message/route.ts`：内联 `createClient()`

### 教训
⚠️ **Supabase JS client 不要在模块级创建**。Next.js 的 `.env.local` 修改需要重启服务器，但即使重启，模块级创建的 client 也可能因热更新不完全而使用旧值。始终在请求处理函数内部创建 client。

## 会话管理功能
（已修复：旧消息按时间分组，不是按单个消息分组）

### 需求
"我的文件"中对话应按会话分批保存，每个对话可点开，左侧恢复聊天历史可继续对话，右侧显示游戏预览。

### 第1次实现的问题
`supabase-add-session.sql` 的 `UPDATE ... SET session_id = gen_random_uuid()` 给每条消息分配了独立 UUID → 每个会话只有 1 条消息。另外 projects 表没有 session_id → 游戏无法关联到会话。

### 修复方案
**不再依赖 DB 的 session_id 列做分组**，改为按时间临近算法：
- 同一用户的连续消息，间隔 ≤ 30 分钟 → 归为同一会话
- 间隔 > 30 分钟 → 新会话
- 游戏关联：寻找创建时间在会话期间内 ±5 分钟的项目

### 当前架构
- **Sessions API**：读所有消息 → 按 30 分钟窗口分组 → 为每组找最近的项目 → 返回 `{session_id, title, message_count, start_time, end_time, game}`
- **Messages API**：支持 `?start_time=xxx&end_time=xxx` 和 `?session_id=xxx` 两种过滤方式
- **loadSession**：优先用 `start_time/end_time`（旧数据），或用 `session_id`（新数据）
- **新消息**：`sendMessage()` 传递 `currentSessionId`，`saveMessage()` 保存到 `session_id` 字段

### 修复：历史会话继续对话无响应

#### 根因
`session_id` 列是 `UUID` 类型，但**历史会话的 sessionId 是时间字符串**（如 `2026-05-14T06:12:53.765859+00:00`）。当从历史会话继续对话时：
1. `currentSessionId` = 时间字符串
2. `saveMessage(..., sessionId)` 试图插入非 UUID 值到 UUID 列
3. PostgreSQL 拒绝，消息保存**静默失败**
4. DeepSeek API 正常返回，但流式数据被正确展示 → 用户看不到错误

#### 修复
1. **`saveMessage()` 增加 UUID 验证**：非有效 UUID 的 sessionId 存为 `null`
2. **`loadSession()` 生成新 UUID**：加载历史会话时，如果 sessionId 不是 UUID，使用 `crypto.randomUUID()` 生成新 ID
3. 时间临近分组机制确保新旧消息仍能正确归入同一会话

| `src/app/api/student/sessions/route.ts` | 时间分组会话 API（核心逻辑） |
| `src/app/api/student/messages/route.ts` | 支持 session_id + 时间范围双模式过滤 |
